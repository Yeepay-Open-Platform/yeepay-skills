# 请求签名协议（出站 API）

> Skill 内语言中立协议参考（不使用官方 SDK 时按此实现）。  
> 权威规则：同目录 `鉴权认证机制(RSA).md`、`鉴权认证机制(SM).md`  
> Python 参考实现：`scripts/rsa/client.py`

## Authorization 头格式（RSA）

```text
YOP-RSA2048-SHA256 yop-auth-v3/{appKey}/{timestamp}/{expireSeconds}/{signedHeaders}/{signature}
```

- `timestamp`：UTC `yyyy-MM-dd'T'HH:mm:ss'Z'`
- `signature`：对 canonical request 做 `SHA256withRSA`，再 URL-safe Base64，后缀 `$SHA256`

## 待签名串（canonical request）

五行，以 `\n` 连接：

```text
{authString}
{HTTP_METHOD}
{canonicalURI}
{canonicalQueryString}
{canonicalHeaders}
```

- `authString` = `yop-auth-v3/{appKey}/{timestamp}/{expireSeconds}`
- **GET**：`canonicalQueryString` = 参数 URL 编码后按 key 升序 `k=v&...`
- **POST Form / POST JSON**：`canonicalQueryString` 为空串
- `canonicalHeaders`：参与签名头（`x-yop-appkey`、`x-yop-request-id`、`x-yop-content-sha256` 等；**不含** `content-type`，见下节）按 key 升序，每行 `urlencode(key):urlencode(value)`

## URL 编码（签名 vs HTTP 报文）

构造签名原文字符串与组装最终 HTTP 报文时，编码规则**不同**，须分两步处理：

| 阶段 | 适用范围 | 编码次数 | 空格 | 说明 |
|------|----------|----------|------|------|
| **签名 canonical** | `canonicalQueryString`、`canonicalHeaders`、POST Form 的 `content-sha256` 输入串 | **一次** | 统一 `%20` | 各语言默认 URL 编码对空格结果不一（`%20` / `+`），签名时必须统一为 `%20`（RFC3986，等同 Java `urlEncodeForSign`） |
| **HTTP 报文** | GET query、POST `application/x-www-form-urlencoded` body | form **值两次**；GET query **值两次、key 一次** | 按 HTTP 协议（可为 `+`） | 解决特殊字符在传输层的兼容问题；等同 Java `URLEncoder.encode` + `UrlEncodedFormEntity`（**不必**把空格统一为 `%20`） |

要点：

- POST Form 的 `x-yop-content-sha256` 对**签名用 canonical 串**（一次编码）计算，**不是**对最终 HTTP body 计算。
- 纯 ASCII 且无 `%`、`+`、`&` 等需转义字符时，两次 HTTP 编码结果与一次相同；含中文、空格、`%` 等时差异明显。
- Python 实现：`common/url_encoding.py`（`url_encode_for_sign` / `http_form_body` / `http_query_string`）。

## x-yop-content-sha256

| 方法 / 类型 | 计算内容 |
|-------------|----------|
| GET | 空串的 SHA256 hex |
| POST Form | 表单 canonical 串的 SHA256 hex |
| POST JSON | JSON body 原文的 SHA256 hex |
| POST multipart | **仅非文件**文本参数的 canonical 串（一次编码）；文件二进制**不参与** |

## Content-Type

| 项 | 说明 |
|----|------|
| GET / POST Form | HTTP 头 `application/x-www-form-urlencoded;charset=UTF-8` |
| POST JSON | HTTP 头 `application/json` |
| POST multipart | **不手写** `Content-Type`（由 HTTP 客户端生成 `multipart/form-data; boundary=...`） |
| 参与签名 | **均不参与**（`Content-Type` 仅作 HTTP 报文头） |
| 接口覆盖 | `build_request(..., content_type=...)` / `call(..., content_type=...)` |
| 实现 | `common/yop_content_type.py` |

## 报文加密（可选）

敏感字段需加密时，增加 `x-yop-encrypt` 等头；RSA 场景为 RSA 加密随机 AES 密钥 + AES 加密报文。详见 `报文加密机制(RSA).md` 与 example 中 `buildEncryptHeader`。

## 处理步骤（语言中立）

```
1. 构造请求参数 / JSON body
2. 计算 content-sha256（POST Form 用**签名用** canonical 串，一次编码），组装参与签名 headers
3. 拼 canonical request 五行（header/form/query 参数均**一次**编码，空格 %20）
4. signature = RSA_SIGN(canonical_request, 商户私钥) → urlsafe_b64 + "$SHA256"
5. Authorization = 协议前缀 + authString + signedHeaders + signature
6. 附加 SDK 标准头（**不参与签名**，见下节）
7. 发送 HTTP 请求：GET query / POST Form body 按**HTTP 协议二次编码**规则组装（见「URL 编码」节）
```

## SDK 标准请求头（不参与签名）

网关根据 `x-yop-sdk-version` 选择**响应报文格式**。未传或版本过低时，易宝可能按旧版 SDK 返回（无 `x-yop-sign-serial-no` 等新特性）。

| 请求头 | 说明 | 脚本默认值 |
|--------|------|------------|
| `x-yop-sdk-version` | SDK 协议版本（固定） | `4.0.0` |
| `x-yop-sdk-langs` | 语言标识 | `python` |
| `x-yop-session-id` | 会话 ID（UUID） | 进程启动时初始化一次，运行期间不变；进程重启后重新生成 |
| `User-Agent` | 运行时标识（**第二段为 Python 版本**，非 SDK 版本） | `python/{python_version}/{os}/{os_release}/` |

实现：`scripts/common/yop_headers.py`，由 `rsa/client.py`、`sm/client.py` 在签名完成后 `update` 到请求头。

## 易错点

| 项 | 说明 |
|----|------|
| URL 编码（签名） | 一次编码，RFC3986，空格 `%20`；与 `urllib.parse.quote(safe="")` / `url_encode_for_sign` 一致 |
| URL 编码（HTTP 报文） | form 值 / GET query 值须 HTTP 编码**两次**；与 `http_form_body` / `http_query_string` 一致 |
| Content-Type | GET/POST Form：`application/x-www-form-urlencoded;charset=UTF-8`；POST JSON：`application/json`；multipart **不设**；**均不参与签名** |
| multipart 文本字段 | 表单项值 HTTP **编码一次**（等同 Java `MultipartEntityBuilder.addTextBody` + `URLEncoder`） |
| multipart 文件字段 | 二进制上传，不参与 content-sha256 |
| 签名 Base64 | **URL-safe**，去掉尾部 `=`，后缀 `$SHA256` |
| POST 查询串 | Form/JSON 的 canonicalQueryString 必须为空 |
| 时钟 | timestamp 与服务器偏差过大导致鉴权失败 |
| SDK 版本头 | 自研客户端须带 `x-yop-sdk-version`（脚本固定 `4.0.0`），否则响应可能为旧格式 |
| 环境 | 沙箱统一网关 `https://sandbox.yeepay.com/yop-center`（**无**独立 yos 域）；生产交易走 `openapi`，**文件上传/下载**走 `yos` |

## 完整示例（固定测试向量，可逐字节比对）

## 文件上传 / 下载（yos 网关）

生产环境 **multipart 上传**与 **GET 下载**均走 yos 网关；沙箱无独立 yos 域，统一走 sandbox 网关。

| 项 | 说明 |
|----|------|
| 生产网关 | 文件上传与下载：`https://yos.yeepay.com/yop-center` |
| 沙箱网关 | **统一** `https://sandbox.yeepay.com/yop-center`（上传、下载与交易相同） |
| 方法 | GET 下载：query 参数按查单规则二次 HTTP 编码；`x-yop-content-sha256` 为空串 SHA256 |
| Content-Type | `application/x-www-form-urlencoded;charset=UTF-8`（不参与签名） |
| 响应体 | 通常为文件二进制流；大文件建议 `stream=True` |

Python：`client.upload(..., sandbox=True)` / `client.download(..., sandbox=True)`；或显式 `base_url=client.DEFAULT_SANDBOX`。生产环境两者默认 yos 网关。下载时 `save_path` 非空写入本地并返回 `Path`，否则返回 `requests.Response`。

### SM 双轨说明

SM2 签名带随机数，**每次结果不同**，不能比对签名值，只能用平台公钥验证「verify 通过」；canonical request 与 content-sha256（SM3）的中间值仍可逐字节比对。

## 应答验签（入站 API 响应）

平台对 API 应答（HTTP body）签名，商户用**平台公钥**验签，与出站请求签名方向相反。

| 项 | 说明 |
|----|------|
| 响应头 | `x-yop-sign`（必填才有验签意义）；SM2 另有 `x-yop-sign-serial-no` 标识平台证书 |
| 待签原文 | 响应 body **去掉所有空格、制表符、换行**后的 UTF-8 字节（`re.sub(r'[ \t\n]', '', body)`） |
| 签名格式 | `{urlSafeBase64Sig}$SHA256`（RSA）或 `{urlSafeBase64Sig}$SM3`（SM2）；亦可能为标准 Base64（含 `+` `/` `=`） |
| RSA 公钥 | 控制台复制的易宝 RSA 公钥；SM2 见 `平台商密证书.md` |
| 无签名头 | 部分接口/环境不返回 `x-yop-sign`，可跳过验签 |

处理步骤：

```
1. 读取 response.text 与 headers['x-yop-sign']
2. normalized = 移除 body 中空格/制表符/换行
3. 解析 x-yop-sign → 签名值 + 摘要算法（$ 分隔）
4. RSA：SHA256withRSA 验签；SM2：SM3withSM2 验签（gmssl verify_with_sm3）
5. SM2 按 x-yop-sign-serial-no 选择平台公钥（多证书缓存）
```

Python 实现：`scripts/common/response_verify.py`；`rsa/client.py` / `sm/client.py` 的 `call(..., verify=True)`；CLI `scripts/tools/verify_response.py`。

应答验签测试向量：`rsa/tests/vectors/response_vector.json`、`sm/tests/vectors/response_sm_vector.json`（`tools/verify_vectors.py` 校验）。
